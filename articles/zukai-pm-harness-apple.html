図解構成 / PM Harness Engineering

PMOをAIに
任せるのではない。

PMOの制御構造を、AIも扱える形にする。原則、危険信号、手順に分解し、最後の線引きは人間が握ります。

Source: Algomatic Tech Blog「PM業務におけるハーネスエンジニアリング」をもとに再構成。

Agent = Model + Harness

モデルの賢さだけでは、自律的に動くAIを制御しきれません。ツール、権限、検証、ログ、実行ルールなど、外側の設計がハーネスです。

Model

賢さ。

LLMそのものの能力。回答し、推論し、ツール利用を選びます。ただし、何を許し、どこで止めるかはモデル単体では担えません。

Agent
= Model + Harness
Harness

外側の設計。

動く前に方向を示し、動いた後に異常を拾い、必要な時に定型手順を呼び出す制御レイヤーです。

FeedforwardGuiderules、前提、判断基準。動く前に方向をそろえる。
FeedbackSensorhooks、検証、危険信号。動いた後の異常を拾う。
ProcedureSkill議事録、WBS、週次報告。必要なときだけ手順を回す。

PMO業務は、人間が回してきた制御システムに近い。

ハーネスの構造をPMOに置き換えると、原則、危険信号、業務手順に対応します。

Guide動く前に判断基準をそろえる。
原則運営方針、リスク管理方針、共有ルール。何を守り、何を守らないかを宣言する。
Sensor動いた後の異常を拾う。
危険信号の検知期限超過、未対応リスク、Decision Drift、Open Question Agingを見張る。
Skill決まった手順を再現する。
業務手順議事録整理、WBS更新、週次報告、リスクチェックを同じ型で回す。

情報の置き場も、ハーネスの一部。

人間が読むものとAIが処理するものを分け、判断基準と検知タイミングをプロジェクト内に置きます。

my-project/
├── CLAUDE.md            # 前提・運営原則
├── .claude/
│   ├── rules/           # 判断基準
│   │   ├── pm-principles.md
│   │   └── risk-management.md
│   ├── hooks/           # 危険信号センサー
│   ├── skills/          # 定型手順
│   └── settings.json    # hooks登録
├── docs/                # 人間向けMarkdown
│   ├── PROJECT.md
│   └── STAKEHOLDER.md
└── state/               # AI向けJSON
    ├── STATUS.json
    ├── RISK.json
    └── WBS.json
Design

docs と state を分ける。

同じ情報でも、人間が判断するために読むものと、AIが安全に読み書きするものは形が違ってよい、という切り分けです。

docs背景、文脈、関係者、方針。人間が読むMarkdown。
state期限、リスク、WBS、状態。AIが検証しやすいJSON。

ミスが起きたら、出力ではなく仕組みを直す。

ハーネスは作って終わりではありません。検知した不具合を改善リストに溜め、rules、hooks、skillsを育て続けます。

Prevent 予防
  • 高リスクは緩和策なしで保存できない
  • 外部送信は人間確認
  • 決定履歴は追記のみ
Detect 検知
  • セッション開始時チェック
  • 期限超過と未回答の問い
  • 方針ズレと未対応リスク
Improve 改善
  • 改善リストに溜める
  • 週次で振り返る
  • ハーネス自体を直す
Core Loop

同じ見落としを、二度起こさない。

人間の注意力ではなく、仕組みの側に学習を移します。PMやコンサルを代替するのではなく、危険信号に気づきやすくする制御レイヤーです。

最後に残る論点は、線引き。

任せられる範囲が広がるほど、どこをAIに委ね、どこを人間が握るかを感覚ではなく仕組みとして決める必要があります。

AIに委ねやすい

  • 期限超過の検知
  • 未回答・未対応の洗い出し
  • 週次レポート下書き
  • リスク一覧の再評価
  • 議事録からの状態更新候補

人間が握る

  • 最後の意思決定
  • 顧客との信頼関係
  • 社内の力学を踏まえた判断
  • 何を検知しないか
  • ハーネス変更そのものの承認
