図解構成 / Codex Rust vs Claude Code Bun

同じ「Nodeなし」でも、
向かう先が違う。

CodexはRustで実行基盤へ。Claude CodeはBunでTypeScriptエコシステムの速さと拡張性へ。

Source: Algomatic Tech Blog「CodexはRustで、Claude CodeはBunなのか」、OpenAI Discussion、Bun/Anthropic公式発表をもとに再構成。

共通の出発点は、AIエージェントの実務化。

ツール実行、ファイル変更、長時間の探索、CLI配布。AIコーディングエージェントが実務で使われ始めると、Node.js前提のままでは扱いにくい課題が見えてきます。

Shared Move

Node.js依存を外す。

導入の壁ユーザー環境にNode.jsがないと動かない。
長時間実行履歴やツール結果を抱えながら動き続ける。
実行制御ファイル、ネットワーク、プロセスを制御したい。
配布性CLIを単体で速く配り、すぐ使えるようにしたい。

分岐点は、Nodeを外した後の作り方。

見た目はどちらも単体実行ファイルへ向かいますが、中身の意味は異なります。

Codex

Rust rewriteで、実行基盤を固める。

CodexはNode/TypeScriptからRustへ寄せ、長時間・多数・安全な実行を支えるsystems寄りの土台を作ろうとしているように見えます。

Rustsandboxingexecpolicynetwork-proxymcp-serverapp-server
runtime GCなし。メモリやレイテンシの見通しを取りやすい。
macOS Seatbelt、Linux Landlock / seccomp などOSの隔離機構に近い。
プロセスツリー全体へsandbox policyを適用する発想。

dependency removed

diverges
Claude Code

Bun executableで、開発者体験を保つ。

Claude CodeはTypeScript周辺のproduct surfaceを保ちながら、Bun executableとしてNode.js runtimeへの依存を外します。

Bun runtimesingle-file executableTypeScriptSDKMCPnpm ecosystem
JS/TSの開発速度、SDK、MCP、npm圏との親和性を維持。
Bun runtimeごと単体実行ファイルにまとめて配布できる。
AnthropicがBunを買収し、Claude Code / Agent SDKのインフラとして投資。

技術選定には、エージェント観が出る。

これは断定ではなく、記事の読み解きです。どちらが上という話ではなく、何を重視しているように見えるかを分けて見ます。

Codex reading 安全に深く、たくさん動かす。 エージェントを、長く走るプロセス群として捉え、sandbox、exec policy、network controlと結びつける方向。
Claude Code reading 速く配り、拡張しやすくする。 エージェントを、開発者が日常的に触るproduct surfaceとして捉え、JS/TS toolingとSDK/MCPの速度を取りにいく方向。
注: Bun executableは単体実行ファイルですが、Rust binaryと同じ意味でruntime GCが消えるわけではありません。ここでの対比は「Nodeを外した後に何を残し、何へ寄せたか」の整理です。
Conclusion

同じ問題に、違う答えを選んだ。

CodexとClaude CodeはどちらもNode.js依存を外しました。ただし、Codexは安全な実行基盤へ、Claude CodeはTypeScriptエコシステムを保った高速な配布・拡張基盤へ向かったように見えます。
