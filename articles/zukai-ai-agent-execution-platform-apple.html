図解構成 / AI Agent Execution Platform

AIエージェントは、
どこで走るのか。

モデルの賢さだけでは足りません。npm install、ファイル変更、外部APIアクセスを許すなら、最後に効くのはOSの境界です。

Source: Algomatic Tech Blog「AIエージェントの実行基盤」をもとに再構成。

その一行は、プロセスになり、カーネル境界を通る。

エージェントが生成したコマンドは、OSから見ると信頼しきれないコードです。CPU、メモリ、ファイル、ネットワークに触れるたび、カーネルがどこまで許すかを決めます。

Agent
AIエージェント「依存を入れてテストして」と判断し、コマンドやコード変更を生成する。
Command
npm install / test / file editshell やランタイム経由で起動し、ファイル、ネットワーク、子プロセスに触れる。
Process
プロセス実行中の最小単位。外界に触れる操作は system call としてカーネルへ届く。
Kernel
カーネルが、走らせ方と止め方を決める。 遅い、kill された、EPERM で弾かれた。原因はモデルではなく、境界設定にあることも多い。
scheduler
CPU割り当て
cgroup
資源上限
namespace
見える世界
seccomp
syscall制限
capabilities
権限分解
遅い
モデルの推論ではなく、CPUスケジューリングや周辺負荷の問題かもしれない。
途中で落ちた
コード不具合ではなく、cgroup 上限と OOM killer の結果かもしれない。
権限エラー
seccomp、namespace、capabilities によって意図的に止められている可能性がある。

実行環境は、host kernel との距離で見る。

container か VM かという名前より、host kernel をどこまで共有するか、syscall・ファイル・ネットワークをどこまで絞るかが本質です。

軽い / 速い / 広い権限隔離が強い / 重い / 小さい攻撃面
host

ほぼ素のホスト

ローカルの自由度は高いが、ファイル、メール、カレンダー、shell など事故時の影響範囲も広い。

  • 速い
  • 便利
  • 権限が広すぎる
container

Container

namespace、cgroup、seccomp、capabilities で囲う。軽い一方、host kernel は共有する。

  • 起動が軽い
  • host kernel共有
  • mount / egress / secret に注意
syscall

gVisor

Sentry が syscall を受け、host kernel へ直接触れる面を減らす。互換性と性能にはトレードオフがある。

  • syscall面を縮小
  • userspace kernel
  • 互換性に注意
microVM

Firecracker

workload ごとに guest kernel と hardware virtualization の境界を置く。serverless 的な使い捨て実行と相性がよい。

  • host kernel非共有
  • 強い隔離
  • containerより重い
OCI + VM

Kata Containers

コンテナの操作感のまま、軽量VMの中で pod や docker run を動かす。Kubernetes と接続しやすい。

  • VM級隔離
  • kubectl と相性
  • VMMを選べる

選ぶべきものは、要件とデータの性質で変わる。

microVM が常に正解でも、container が常に危険でもありません。見るべきは、何を許し、どこで止め、何を記録するかです。

host kernel共有するのか、guest kernel と VM 境界を置くのか。
filesystemどのディレクトリを見せ、書き込みをどこまで許すのか。
network egress外部API、package registry、任意URLへの通信をどう絞るのか。
secrets秘密情報を環境内に露出させるのか、brokering するのか。
auditコマンド、ファイル差分、通信、失敗理由をどの粒度で残すのか。
cost / latency起動速度、互換性、隔離強度、運用コストをどう天秤にかけるのか。
Conclusion

新しいのは、相手がエージェントになったこと。

効く部品は、プロセスを分け、資源に上限を置き、system call を絞り、ネットワークの向きを決めるという、OSが昔から持ってきた境界です。
