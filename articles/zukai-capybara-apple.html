図解構成 / Capability-independent Codebase

注意力に依存しない
コードベース。

AIも人間も間違えます。だから、良い実装を期待するのではなく、悪い状態を作れない入口と型を設計します。

Source: Algomatic Tech Blog「実装者の能力に依存しないコードベース」をもとに再構成。

「気をつけてね」は、設計ではない。

手順を呼び出し側が毎回覚えるコードでは、1行抜けただけで業務上不完全な状態ができます。

Bad

呼び出し側が、毎回覚える。

注文を作ったら、保存し、監査ログも忘れずに書く。複数箇所に広がるほど、注意力への依存が増えます。

const order = Order.create(input) await orderRepository.save(order) await auditLogRepository.record(order)
Good

業務操作として、閉じる。

必要な手順を placeOrder の内部に閉じます。利用側は個別手順を知らず、正しい入口だけを使います。

await placeOrder({ customerId, items, requestedBy, })
防ぎたい状態: 注文は存在するのに、監査ログがない。これはレビューで見つけるより、構造で作れなくする方が強い。

入口と型で、不正状態を閉じ込める。

解き方は文脈次第ですが、共通する狙いは「呼び出し側の自由な組み合わせ」を減らすことです。

Operation

業務操作に名前を付ける。

create / save / audit のような個別手順を外へ漏らさず、業務上意味のある関数を入口にします。

export async function placeOrder(input) { const order = createOrder(input) await saveOrder(order) await recordOrderAuditLog(order) return order }
内部処理は外部モジュールから直接呼べないようにする。
イベント駆動で「注文作成」を起点に監査ログを出す選択肢もある。
Type / Factory

ありえない組み合わせを作れなくする。

email 通知なのに phoneNumber を渡せるような状態を、型や factory でビルド前に止めます。

sendNotification(emailNotification({ to: emailAddress, subject, body, })) sendNotification(smsNotification({ to: phoneNumber, body, }))
通知チャネルと宛先の対応関係を呼び出し側に任せない。
イベント名と種別、外部サービスと認証方式、状態と操作にも同じ考え方を使う。

AIコーディング時代ほど、この原則が効く。

AI固有の問題ではありません。信用できない実装者と働く前提では、人間でもAIでも、間違えようがない設計が必要です。

Weakness AIは間違える。 仕様を取り違えたり、呼び出し側の手順を抜かしたりします。だから、レビュー頼みではなく構造で止めます。
Strength AIは徹底できる。 人間が面倒で崩しがちな厳密なルールも、疲れずに守れます。だから、設計をコードベースに埋め込みます。
Conclusion

レビューで見つけるより、そもそも作れない形にする。

良いコードを書くことを実装者に期待するのではなく、悪い状態を作れない入口、型、factory、イベント境界をコードベースに置きます。
