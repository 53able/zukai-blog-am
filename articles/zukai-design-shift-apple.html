デザイナーは、
翻訳者ではなくなるのか。

Figma中心の翻訳作業から、AIによって実物検証へ移り、デザイナーが本来の相手であるユーザーへ向き合いやすくなる。

Source: Algomatic Tech Blog「デザイナーは翻訳者ではなくなるのか」をもとに再構成。

これまでは、実物の前に翻訳があった。

Figmaは目的ではなく、意図を次の作業者へ伝えるための手段でした。実装コストが高いほど、翻訳の精度が重要になりました。

1 体験を考える。 業務、情報設計、画面状態、ユーザーの行動を頭の中で組み立てます。
2 Figmaへ翻訳する。 余白、状態、コンポーネント、補足説明を、伝わる成果物に変換します。
3 実装へ渡す。 エンジニアやチームが、絵と仕様から動くプロダクトへ戻します。
4 ユーザーに届く。 本当に確かめたい相手へ届くまでに、何度も変換が挟まります。

AIが下げたのは、実装コストだけではない。

曖昧な意図から動くものを作れるようになると、伝えるために作り込む成果物の量が減ります。翻訳の手前で、実物に触れられるようになります。

Before 絵を精密にして、認識ズレを減らす。 動くものをすぐ確かめられないため、Figmaの中に状態・仕様・注釈を詰め込みました。
デザインQAページを作る
スクリーンショットとカンプを並べる
修正指示を細かく伝える
After 実物を触って、ズレを早く見つける。 AIでプロトタイプやプロダクションコードに近づけると、伝達物ではなく実物で議論できます。

ツールの歴史は、伝わる距離を縮めてきた。

Figma不要論ではありません。デザインツールは、ずっと「より正確に伝える」方向へ進化してきました。AI以後は、その先にある実物へ重心が動きます。

Draw Illustrator / Photoshop 見た目を伝える。Webやプロダクトの動きまでは、別の言葉で補う必要がありました。
Motion Prott / ProtoPie 画面遷移や動きを伝えやすくし、静止画だけでは伝わらない体験を補いました。
System Sketch / XD / Figma AutoLayout、Components、Variantsで、実装に近い構造まで伝えられるようになりました。
AI 実物プロトタイプ ラフなUIや意図から、触れるものを短時間で作り、検証の単位を前倒しできます。
Code Figma Code layers Figma自身も、絵を扱う場所から実物に近いものを扱う場所へ動こうとしています。

翻訳先が減ると、検証先が近くなる。

成果物の作り込みが消えるのではなく、必要なときに必要な粒度で使う形へ変わります。主役は、触れる実物とユーザーの反応です。

Prototype
as the Message 説明用の絵から、触って分かる実物へ。伝える媒体そのものが変わります。
Design QA 指示を書く前に直す。 フロントエンドを自分で修正できると、翻訳すべき相手が減ります。
Website 一画面でトンマナを決める。 全ページを作り込む代わりに、必要な要素を都度AIへ渡します。
Product 1〜2日で触れる形へ。 開発チームやユーザーに触ってもらい、抜け漏れを早く見つけます。

なくなるのは、デザインではない。

減っていくのは、中間に存在した制作上の翻訳作業です。デザイナーの向き合う先は、よりユーザーへ近づきます。

Keep Figmaは、必要な粒度で使う。 絵で見せるほうが速い部分、構造を共有したい部分では、これからも有効な道具です。
Shift 実物で、ユーザーに近づく。 作ることと伝えることの境界が融けるほど、本当に伝えるべき相手へ時間を使えます。
Conclusion

翻訳者から、検証者へ。

AIはデザイナーを消すのではなく、翻訳に追われていた時間を、ユーザーと実物に向け直します。
