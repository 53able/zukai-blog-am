評価は、足場を揃えてから行う。

モデルの性能差を見るために、タスクの作成、レビュー、実行、採点、公開までの流れを固定します。

01新規タスク作成既存コミットや公開PRの答えを使わない。
02人間レビュープロンプト、検証器、参照実装を確認する。
03mini-swe-agent各モデルに同じ bash ツールと共通プロンプトを渡す。
04Verifier公開APIや出力など、外から見える振る舞いを採点する。
05公開データスコア、コスト、時間、軌跡を確認できる。

タスク例は、単発の修正ではなく設計を含む。

記事では、CSS lexer と JSON Schema parser の2例から、正確な実装力と構成力の違いを見ています。

precision

CSS shorthand を展開・圧縮する

background の複数レイヤー、longhand への分配、最終レイヤーだけに適用される background-color など、細かい仕様を正確に扱う必要があります。

architecture

JSON Schema の $ref と依存関係を扱う

既存パーサーの暗黙 object schema、再帰参照、anyOf 合成などを踏まえて、局所修正だけでなく全体の構成を整える必要があります。

注意: DeepSWE も完全な評価ではありません。公式ブログは、Java/C++未対応、OSS中心、mini-swe-agent固定、bug localization や refactoring の不足などを限界として挙げています。図では「信頼できる絶対評価」ではなく、「比較の解像度を上げる仕組み」として扱います。
